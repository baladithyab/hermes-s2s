## 7 · MetaCommandSink false positives — **P1 FAIL**

Pattern at `meta.py:137`: `^(?:stop|cancel|hush|quiet)\b.*$`. The `.*$`
tail greedily consumes anything. `"hey aria stop the music"` matches
`stop_speaking` with `consumed=True`, meaning:

1. TTS is interrupted (side-effect the user didn't ask for).
2. The utterance is **dropped** and never reaches the LLM, so the Spotify
   pause the user actually wanted never happens.

**The other patterns are fine** — `new`, `compress`, `title`, `branch`,
`clear` all require specific noun phrases (`new session`, `the context`,
etc.) so they don't false-fire. `stop_speaking` is the only over-broad verb.

**Fix:** tighten to `^(?:stop|cancel|hush|quiet)\s*(?:speaking|talking|it)?\s*$`
or require no trailing object — i.e. allow terminal punctuation/whitespace
only. Existing test suite should add an adversarial case.

Mental replay of the task's other cases: ✅ `"hey aria please start a new
session about React"` → filler `please` stripped, `new` matches with
`extra="about React"` (via `\b(?P<extra>.{0,80})` and the final `$`). ✅
`"I should start a new feature"` → no wakeword, returns None.

## 8 · Migration sentinel + multi-machine sync — **P1 FAIL**

Flow in `config/__init__.py:114–160`:
1. `s2s.voice.default_mode` present → already migrated, return.
2. `s2s.mode` missing → nothing to migrate, return.
3. Sentinel `<hermes_home>/.s2s_migrated_0_4` exists → **return unmigrated**.

**The bug:** in step 3, the code has already established that the config IS
legacy (step 1 failed AND step 2 found `s2s.mode`). Returning early leaves
the user's runtime pointing at a half-invalid config where downstream
consumers silently default to `cascaded` mode, masking whatever the user
configured in 0.3.x.

Scenario: Syncthing / Dropbox / iCloud syncs `~/.hermes/` between laptop A
and desktop B. A migrates (sentinel + rewritten config.yaml). B receives the
sentinel but `config.yaml` is edit-conflict-quarantined (e.g. `.sync-conflict`
file) OR simply hasn't synced yet due to differential timing. B starts
hermes-s2s → sentinel exists → auto-translate skipped → user sees mode
reset to default.

**Fix (two options, either works):**
- (A) Remove the early-return on sentinel when `has_legacy_mode=True`
  — the sentinel is an *optimization* to skip file I/O, not a correctness
  gate. If there's still a legacy key, migrate regardless.
- (B) Key the sentinel on a content hash of the translated config, so a
  mismatched sentinel forces re-migration.

(A) is simpler and matches the intent.

## Verdict: **FIX-AND-RETAG**

The P0 at finding 5 is a functional regression dressed up as a passing
test. The voice bot will ship with a critically pruned tool manifest —
no `todo`, no `skills_list`, no HA reads, no browser reads. On a normal
user session this will manifest as the bot apologetically declining
half of its nominal capabilities. The CI fence that was *supposed* to
prevent exactly this scenario is itself the source of the lie.

Minimum changes to clear tag push:

1. Replace `_CORE_TOOLS_AT_0_4_0` with a runtime import of
   `_HERMES_CORE_TOOLS` (or a build-time copied snapshot generated from
   it). Fail if the snapshot goes stale.
2. Classify the 12 unknown tools into DEFAULT_EXPOSED / ASK / DENY.
3. Re-run `test_tool_export.py` — it should still pass but now on
   reality.

P1 items (7, 8) are recoverable in a 0.4.1 patch tag. P0 (5) is not — it
would require a 0.4.0.post1 within 48 hours of any user report.

**Do not push the tag until finding 5 is addressed.**
