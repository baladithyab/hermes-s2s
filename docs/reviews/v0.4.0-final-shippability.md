# v0.4.0 Final Shippability Review

**Scope:** Phase-8 whole-branch review of `feat/0.4.0-rearchitecture` @ `ab5ee41`
(version bump W6b, 6 batches landed) through a fresh-user lens.
**Lens:** Does this *actually* meet ASK1–ASK5, and can a fresh installer get
past a first `/voice join` without reading code?
**Caveat:** Route-fidelity RED, Opus-on-Opus — treat as internal eyeball, not
independent verification.

**Verdict:** 🟡 **FIX-AND-RETAG.** The skeleton is there and ships, but
three user-visible regressions/mismatches will show up inside 60 seconds
of a fresh user hitting the bot. All three are single-commit fixes.

## Fresh-user walkthrough

### 1. Install path — 🟡 mostly correct

README's 30-second install is accurate (`hermes plugins install …` → venv
`pip install -e '[all]'` → `hermes plugins enable` → `hermes s2s setup
realtime-gemini` → `hermes s2s doctor`). Extras are up to date. No new
system libs required beyond 0.3.x (ffmpeg, espeak-ng for Kokoro,
libopus). **Gap:** README doesn't mention the `HERMES_S2S_MONKEYPATCH_DISCORD=1`
env var is still required for `/s2s` to install — the wizard writes it
automatically for realtime profiles but a user picking `cascaded` + wanting
the slash command loses it silently.

### 2. First-run setup — 🟡 wizard is fine, migration is broken

`hermes s2s setup` flow is unchanged from 0.3.x, which is good — but it
doesn't surface the **four modes** concept at all. A user coming from
docs does not see a "pick a mode" step; they see "pick a profile," and
profiles happen to imply modes. Migration from 0.3.x is silent
auto-translate on first `load_config`, *and* this path has a bug (Issue
#3). The non-destructive merge in `cmd_setup` (M5.2) is correct.

### 3. First `/voice join` — 🟢 acceptable

With a `realtime-gemini` profile + `HERMES_S2S_MONKEYPATCH_DISCORD=1`,
the bridge attaches, thread auto-creates on a plain channel (ThreadResolver
with 1-day auto-archive, forum parents fall back). Starter message posts.
Transcripts mirror via `TranscriptMirror` with token-bucket + bounded
overflow queue. **Missing:** no privacy/security warning in the starter
message that a public thread is being created and transcripts will be
mirrored to it. Recommended but not blocking.

### 4. Discoverability of `/s2s` — 🟡 Choice labels are OK, behaviour doesn't match docs

`slash.py:396-410` labels are user-friendly:

- "Cascaded (default Hermes voice)"
- "Pipeline (custom STT+TTS)"
- "Realtime (Gemini Live / GPT-4o)"
- "External S2S server"

Good. **BUT** `mode` is a **required** parameter. There is no `/s2s`
no-arg invocation. There is no `/s2s test`. The HOWTO (§0.4.0 `/s2s`
block, lines 36–42) and the README §What's-new (line 18:
`/s2s test hello there`) both advertise these. **Either the docs lie
or the command is half-built.**

### 5. Voice meta-command discoverability — 🔴 broken out of the box

With zero docs, a user will guess "hey aria" (default wakeword in code).
With the docs, a user will try "hermes new" — which gets no match
because (a) wakeword is `hey aria`, not `hermes`; (b) the grammar
requires phrases, not verbs. See Issue #1.

### 6. Documentation completeness — 🟡 mixed

| Topic | Covered? |
|---|---|
| per-VC mode selection | ✅ HOWTO §0.4.0 `/s2s` block |
| thread mirroring | ✅ HOWTO §Thread mirroring, README What's-new |
| privacy note on public threads | ❌ absent |
| meta commands | 🔴 documented with **wrong wakeword + wrong grammar** |
| migration from 0.3.x | ✅ README + HOWTO + dedicated `migrate_0_4.py` header |
| prompt-injection defense | ✅ HOWTO §Voice persona overlay |
| error when API key missing | ⚠ doctor explains generally, but no "what you'll see in Discord" snippet |

### 7. Error UX (missing `GEMINI_API_KEY`) — 🟢 good

`capabilities.CapabilityError.user_message()` returns:

> ❌ Voice mode **realtime** needs: `env:GEMINI_API_KEY`. Set it up or pick a different mode.

…and `discord_bridge._handle_capability_error_rollback()` disconnects
the VC and posts that message into the bound text channel. This is the
right UX. One nit: the `env:` / `pip:` prefix leaks implementation
detail into Discord ("needs: `env:GEMINI_API_KEY`"). A fresh user won't
know `env:` is a tag — recommend stripping to just "needs: `GEMINI_API_KEY`".

### 8. Migration smoke test — 🔴 **translates value incorrectly**

I ran `migrate_0_4 --dry-run` against a fixture 0.3.x `mode: duplex`
config. Diff output:

```diff
+  voice:
+    default_mode: duplex           # <-- still says "duplex"
+    wakeword: hey aria
+    thread_name_template: 🎤 {user} — {date:%Y-%m-%d %H:%M}
```

**`duplex` is not a valid 0.4.0 `VoiceMode`.** `VoiceMode.normalize("duplex")`
raises `ValueError`. README §0.4.0 claims "the legacy `s2s.mode: duplex`
boolean is silently auto-translated to `s2s.mode: realtime` on first
load" — but `translate_config` in `migrate_0_4.py:128-131` just copies
the string. A 0.3.x user with `mode: duplex` gets a broken 0.4.0 config
that fails at `ModeRouter.resolve()`. This is the single most embarrassing
fresh-install bug in the branch.

### 9. `/voice leave` UX — 🟡 silent

`_wrap_leave_voice_channel` cleans up the bridge, but nothing posts a
"session ended" message in the thread. For 0.4.0 this is acceptable
(matches 0.3.x behavior); recommend adding a single-line "🔇 Voice session
ended" to the thread for 0.4.1.

### 10. Other gaps for shippability

- CLI `hermes s2s mode` accepts only `["cascaded", "realtime", "s2s-server"]`
  — **missing `pipeline`** (cli.py:317). Three-mode world leaked through.
- Version mismatch between migrator default wakeword (`"hey aria"`) and
  README (`"hermes"`).
- No `s2s.voice.thread_privacy_notice` config key; thread mirror is
  silent-by-default even though it's public.

## Issues (at least 3, per brief)

### 🔴 Issue #1 — Meta-command docs vs. code diverge completely

- **Code:** wakeword default is `"hey aria"` (meta.py:80,
  migrate_0_4.py:57); grammar is full phrases (`"start a new session"`,
  `"compress the context"`, `"title this session as X"`).
- **Docs:** README/HOWTO tell users the wakeword is `"hermes"` and give
  examples `hermes new`, `hermes compress`, `hermes title`, `hermes branch`,
  `hermes stop`, `hermes clear`. **None of these match the grammar.**
- **Impact:** ASK4 effectively ships broken from a user perspective. A
  user who reads docs and speaks those commands gets silent no-ops.
- **Fix options:** (a) add alias grammar `^(new|compress|title|branch|
  stop|clear)$` so single-word verbs match, OR (b) change the
  default wakeword to `"hermes"` AND add single-word verb alternation.
  Option (b) matches the docs. 1-commit change in `voice/meta.py` +
  `migrate_0_4.py:DEFAULT_WAKEWORD`.

### 🔴 Issue #2 — README/HOWTO advertise `/s2s` subcommands that don't exist

- **Code:** `slash.py` registers exactly ONE command with a REQUIRED
  `mode: Choice[str]` arg. No `status` branch. No `test` branch.
- **Docs:** HOWTO lines 36–42 and README line 18 show `/s2s` (no args,
  "show current mode + readiness"), `/s2s test` and `/s2s test "hello
  there"`.
- **Impact:** ASK2 — user tries `/s2s` or `/s2s test`, Discord rejects
  it as missing required arg, the "commands UX" story is visibly broken
  at first contact.
- **Fix options:** (a) implement as a command group with `mode` / `status`
  / `test` subcommands (1–2 hrs of work), OR (b) delete those examples
  from docs and ship the single-command form. (a) honors the advertised
  surface.

### 🔴 Issue #3 — `migrate_0_4` does NOT translate `duplex → realtime`

- **Code:** `translate_config` preserves the old value verbatim
  (migrate_0_4.py:128-131).
- **Docs:** README §0.4.0 says the legacy boolean "is silently
  auto-translated to `s2s.mode: realtime` on first load".
- **Impact:** Every 0.3.x user with the default `mode: duplex` gets a
  0.4.0 config whose `default_mode` value `VoiceMode.normalize()`
  rejects with `ValueError`. ModeRouter raises on every resolve.
- **Fix:** 3-line change — if `old_mode in {"duplex"}`, set
  `voice["default_mode"] = "realtime"` and append a specific warning.
  Also add `"cascaded"` as the fallback for unknown legacy values.

### 🟡 Issue #4 — ASK3 framing mismatch

User asked for "voice/s2s to auto-co-manage threads with hermes
gateway/core". ADR-0012 titled "voice-thread-co-management" is honest
that this is a plugin-side runner monkey-patch of
`AIAgentRunner._handle_voice_channel_join` with no upstream PR filed. The
README reads like it's first-class. Expect user pushback on the word
"co-management". Lowest-effort fix: rename the README bullet to
"plugin-side thread management (upstream PR pending)" and file a tracking
issue in hermes-agent.

### 🟡 Issue #5 — CLI `hermes s2s mode` missing `pipeline`

`cli.py:317` — `choices=["cascaded", "realtime", "s2s-server"]`. Four
modes everywhere else, three here. 1-line fix.
