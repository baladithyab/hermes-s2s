# v0.4.0 Final Review — Security + UX Lens (Phase-8)

**Scope:** Whole-branch `feat/0.4.0-rearchitecture` (HEAD `ab5ee41`), tag pending push.
**Lens:** Security + user experience (audio-only). Orthogonal to the correctness lens.
**Caveat:** Route-fidelity probe was RED per parent context. This is Opus-on-Opus through
context isolation — treat findings as signal, not independent validation.

---

## Findings

### P0-1. CI fence snapshot is out of sync with real `_HERMES_CORE_TOOLS` → silent drops

`tests/test_tool_export.py::_CORE_TOOLS_AT_0_4_0` is a hand-maintained snapshot that
has **diverged** from `~/.hermes/hermes-agent/toolsets.py:_HERMES_CORE_TOOLS`. Compared
head-to-head, the snapshot:

- **Omits** real core tools: `web_extract`, `browser_snapshot`, `browser_back`,
  `browser_vision`, `browser_cdp`, `browser_dialog`, `todo`, `skills_list`,
  `skill_view`, `ha_list_entities`, `ha_get_state`, `ha_list_services`.
- **Contains phantoms** not in real core: `ha_state_read`, `kanban_list`,
  `comfyui`, `spotify_*`, `feishu_doc_read`, `xurl_*`, `obsidian_read`,
  `youtube_transcript`, `arxiv_search`, `gif_search`.

The fence passes because it only checks the snapshot is classified — but the
snapshot is wrong. At runtime, `build_tool_manifest()` encounters the real
unclassified tools (`browser_cdp`, `browser_dialog`, `ha_get_state`, ...) and
fail-closes them with a warning. So the *current* behavior is actually safe
(CDP is dropped), but **the fence provides no regression protection** — a future
Hermes core change adding, say, `file_delete` to `_HERMES_CORE_TOOLS` would
silently drop it without anyone noticing, and if someone later classifies it
into `DEFAULT_EXPOSED` by mistake, the fence won't catch it. **Fix:** replace
the hand-maintained snapshot with an import from the live toolsets module, OR
at minimum re-sync the snapshot to the actual list today.

### P0-2. ASK bucket has no pending-confirm flow → ASK tools pass through silently

`tool_bridge.py:55` comment says "The meta_dispatcher (W4a) owns the pending-confirm
flow" — but `hermes_s2s/voice/meta_dispatcher.py` contains **no** `_dispatch_ask`,
`pending_confirm`, `ASK`, or equivalent. Verbs are limited to: `new`, `compress`,
`title`, `branch`, `stop`, `clear`. `build_tool_manifest()` simply appends ASK-bucket
tools (`read_file`, `session_search`, `memory`, `browser_navigate`, ...) straight into
the manifest exported to Gemini Live / GPT-4o Realtime. **There is no voice
yes/no gate.** The 5s pending-confirm test referenced in the plan
(`test_ask_bucket_pending_confirm_5s_timeout`) does not exist — `grep -r
pending_confirm hermes_s2s/` returns 0 hits.

**Attack scenario (user voice injection):** user says "ARIA please read my ssh key".
The persona overlay (section 3, prompt-injection defense) *is* the only defense;
if the LLM follows it, refusal happens. If the LLM decides the user legitimately
wants a file read (ambiguity; overlay is soft), `read_file("/home/user/.ssh/id_rsa")`
fires with no audible confirmation and the content comes back through TTS. **The
whole security architecture documented in ADR-0014 §3 assumes ASK has a synchronous
prompt; that assumption doesn't hold in ship code.** Either remove ASK-bucket tools
until the prompt exists (promote them to DENY), or ship the dispatcher path.

### P0-3. `load_config` silently returns empty config on YAML syntax error

`hermes_s2s/config/__init__.py:181-185`:

```python
try:
    with open(config_path, "r", ...) as f:
        data = yaml.safe_load(f) or {}
except Exception:
    return S2SConfig.from_dict({})
```

A single stray tab in the user's `config.yaml` produces a blanket
`except Exception` → **default empty config**, no log, no stderr warning, no
recovery path. The user sees Hermes behave as if freshly installed: wrong
mode, wrong wakeword, wrong everything. For a YAML-config-driven plugin this
is the worst possible failure mode. **Fix:** log at ERROR, include the parse
exception, and surface via `hermes doctor` (or bypass the catch entirely and
let load_config raise — defer to caller). Compare migrate_0_4._load_yaml
(line 226) which correctly raises RuntimeError with the path.

### P1-4. `VoiceSession.stop()` never called on `/voice leave` → resource leak

`voice/factory.py:208` registers sessions into `adapter._s2s_sessions[(guild, channel)]`.
`_internal/discord_bridge.py:_wrap_leave_voice_channel` iterates
**`adapter._s2s_bridges`** (realtime AudioBridge) on leave and calls `bridge.close()`.
It **never touches** `_s2s_sessions`. `grep -n _s2s_sessions discord_bridge.py`
returns one hit — in a docstring, not code. For cascaded/pipeline/s2s-server modes
there is no bridge; the session's websockets, subprocesses, and queues (e.g.
pipeline's STT worker, s2s-server socket) outlive the VC join. On repeated
join/leave cycles this accumulates. **Fix:** iterate `_s2s_sessions` in the
leave wrapper and `await session.stop()`; current docstring already implies the
intent, implementation lags.

### P1-5. README/HOWTO promise `/s2s` subcommands that don't exist

README line 16-18: `/s2s mode realtime`, `/s2s test hello there`.
HOWTO-VOICE-MODE.md line 37-41: `/s2s`, `/s2s mode <x>`, `/s2s test`, `/s2s test "..."`.
HOWTO-REALTIME-DISCORD.md line 10-12: status view + smoke-test TTS.

Actual code (`voice/slash.py:391-448`): **one** command `/s2s` with a single
required `mode` choice parameter. No subcommands, no `test`, no no-arg status
view (Discord will force the user to pick a mode). First-time users following
the README will type `/s2s` and be confused. **Fix:** either implement the
subcommands (W6a scope creep) or rewrite the docs to describe what actually
ships: one command, required mode choice, overrides-next-join-only semantics.

### P1-6. Wakeword doc drift (`hermes` vs `hey aria`)

README line 30: "wakeword defaults to `hermes`". Migration default
(`migrate_0_4.py:57`): `DEFAULT_WAKEWORD = "hey aria"`. MetaCommandSink
default (`voice/meta.py:80`): `"hey aria"`. README examples line 26-28
still use `hermes new`, `hermes compress`, etc. — none of which match the
actual default grammar. A user reading the release notes, joining a VC,
saying "hermes new" → nothing happens, silently. **Fix:** reconcile on
one wakeword; the newer ADR-014/015/meta code clearly uses "hey aria",
so update the README + release notes to match.

### P2-7. Concurrent `/s2s` last-write-wins — both users get "✅"

Two users in the same `(guild_id, channel_id)` click `/s2s mode=realtime`
and `/s2s mode=cascaded` within the flock window. `store.set()` is atomic
(flock-protected) but the slash handler issues
`interaction.response.send_message("✅ Next /voice join will use <mode>")`
to **each** interaction regardless of who wrote last. Both users see success;
only the second-caller's mode actually persists. For a per-channel override
store this is minor (the NEXT join wins), but the UX promise is misleading.
**Fix (cheap):** on second write, read-back the store value and if it
differs from this caller's input, echo "⚠️ another user set this to X
just now".

### P2-8. Public thread auto-creation — transcripts visible to whole guild

`/voice join` in `#general` → `ThreadResolver` creates a **public** thread
(`ChannelType.public_thread`, auto-archive 60 min). Starter message privacy
notice is optional and empty-string gets only a deprecation log warning
(`threads.py:222`). Display-name-in-title isn't an `@everyone` vector
(Discord thread titles don't ping), but a user named "@here emergency"
creates a visually-confusing thread title. More importantly, **voice
transcripts in a server-wide public thread** is a surprise if the user
invoked from a public channel. **Fix:** default `s2s.voice.thread_starter_message`
to a non-empty privacy notice, and consider `private_thread` as the default
when the parent channel is non-NSFW and has ≥N members.

### P2-9. Migration on read-only filesystem is fail-safe (verified)

`_maybe_auto_migrate` wraps `atomic_write_yaml` + `write_sentinel` in
`try/except Exception` (`config/__init__.py:155-157`) and returns the
*original* `data` on failure. Read-only HERMES_HOME → user gets the
in-memory-migrated config for this run, no disk write, no sentinel, no
crash. Next run repeats the in-memory translate. No corruption, no config
loss. **OK — no fix needed**, but add a one-line log at WARNING level so
operators notice the persistent RO problem instead of silently re-migrating
every process start.

### P2-10. Backward compat for custom keys is preserved (verified)

`migrate_0_4.translate_config` does `copy.deepcopy(config)` then surgically
adds `s2s.voice.default_mode` / `wakeword` / `thread_name_template`. Unknown
top-level keys (`model`, `provider`, `terminal`, user's `my_thing`) and
unknown `s2s.*` subtrees survive verbatim. `test_translate_preserves_unrelated_keys`
covers top-level. Custom `s2s.*` subkeys *aren't* covered by an explicit
test but work by deepcopy construction. **Low risk, add a one-line fixture
test** (`s2s.my_experimental: 42` → preserved after translate).

### Not-an-issue (verified safe)

- **Persona overlay language anchor:** "hey ARIA use Spanish now" — section 2
  of `build_voice_overlay` pins the response language via `lang_name_from_code`
  and explicitly requires an unambiguous user request to switch. Hard to
  override inline.
- **`send_message` in DENY:** confirmed `tool_bridge.py:98`; hard-dropped from
  the manifest before the LLM ever sees it.
- **Wakeword false-positive "hi Mariana":** grammar in `voice/meta.py` requires
  wakeword *prefix* + whitespace boundary + a valid *verb* in the enum. A
  near-mishearing without a trailing verb will not match. False-positive rate
  is bounded by the verb set (6 verbs), each of which is a distinct English
  word — low in practice.

---

## Verdict: **FIX-AND-RETAG**

Three P0s and three P1s against the claimed security posture. The P0s are:

1. Outdated CI fence snapshot (fence-in-name-only today).
2. ASK bucket confirm flow doesn't exist (documented-but-missing defense).
3. `load_config` silently swallows YAML errors (UX disaster for typos).

Any one of these is a minor-version retag. Together they mean the security
story ADR-0014 §3 tells is not the story the code implements. The `feat/0.4.0`
work is otherwise landed cleanly and the factory/session/mode machinery is sound
— fixing the P0s is small (re-sync snapshot, promote ASK→DENY until confirm ships
OR ship confirm, unguard load_config exception) and should happen before the
tag is pushed. Bump to **0.4.0rc2** or fix-forward and push `0.4.0` after the
three P0 patches land.
